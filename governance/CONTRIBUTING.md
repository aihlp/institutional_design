# Contributing to Institutional Design Lab

Thank you for your interest in contributing to the Institutional Design Lab! This repository is a collaborative research infrastructure for studying institutional dynamics in digital systems.

---

## Quick Start

1. **Fork** the repository
2. **Create a branch** for your contribution (`git checkout -b feature/your-feature`)
3. **Make changes** following our guidelines below
4. **Test** your changes locally if applicable
5. **Submit a Pull Request** with clear description

## Contribution Guidelines

### Code Quality

- Write clear, documented code
- Follow existing style conventions
- Include tests for new functionality
- Document dependencies

### Documentation Standards

- Use Markdown for all documentation
- Include inline citations where appropriate
- Link to related concepts using `[[wiki-style]]` links
- Maintain consistent terminology from ontology

### Theory Contributions

When adding theoretical content:

1. **Ground in literature**: Cite relevant academic sources
2. **Connect to ontology**: Link to existing concepts
3. **Specify scope**: Clarify boundary conditions
4. **Enable operationalization**: Suggest measurement approaches

### Dataset Contributions

When adding dataset information:

1. **Document provenance**: Source, collection method, time period
2. **Specify schema**: Structure, variables, types
3. **Note limitations**: Biases, gaps, representativeness concerns
4. **Include access info**: How to obtain, any restrictions

### Simulation Contributions

When adding simulations:

1. **Document assumptions**: Model specifications clearly
2. **Provide parameters**: Configurable settings documented
3. **Include validation**: Comparison to empirical patterns if possible
4. **Enable reproduction**: Seed values, environment specs

## Pull Request Process

### Before Submitting

- [ ] Code follows existing style
- [ ] Tests pass (if applicable)
- [ ] Documentation updated
- [ ] No sensitive data included
- [ ] Ethics considerations addressed

### PR Description Template

```markdown
## Summary
[Brief description of changes]

## Motivation
[Why this change is needed]

## Changes
- [List of specific changes]

## Testing
[How you tested the changes]

## Related Issues
[Links to related issues]

## Ethics Considerations
[Any ethical implications addressed]
```

### Review Process

1. **Automated checks**: CI/CD pipelines run
2. **Maintainer review**: At least one maintainer reviews
3. **Community input**: Significant changes open for community comment
4. **Approval & merge**: Maintainer approves and merges

## Communication

### GitHub Issues

Use for:
- Bug reports
- Feature requests
- Specific technical questions

Tag appropriately:
- `bug` - Something isn't working
- `enhancement` - New feature or improvement
- `documentation` - Documentation improvements
- `ethics` - Ethical considerations
- `theory` - Theoretical discussions
- `simulations` - Simulation-related
- `datasets` - Dataset-related

### GitHub Discussions

Use for:
- General questions
- Research direction proposals
- Community coordination
- Show and tell

### Code of Conduct

Be respectful and inclusive:
- Focus on ideas, not individuals
- Assume good faith
- Welcome diverse perspectives
- No harassment or discrimination

See [GOVERNANCE.md](governance/GOVERNANCE.md) for full details.

---

## Ethics Review

Contributions that raise ethical concerns undergo additional review:

1. Contributor flags potential concerns in PR
2. Maintainers assess severity
3. Community consultation if significant
4. Decision documented transparently

See [ETHICS.md](governance/ETHICS.md) for ethical framework.
